# Спека: свободные руки во время обработки (возврат фокуса, ETA, Esc)

Дата: 2026-07-20

## Проблема

Во время распознавания (~10 с на минуту аудио) пользователь обязан держать
фокус в целевом поле: Ctrl+V уходит в окно, активное **в момент окончания**
обработки. Переключился — текст улетает не туда (v1.4.0 спасает его через
трей, но вставку не чинит). Фидбек: «В этот момент нельзя ничего делать,
чтобы не потерялась информация о том, что было сказано»; предложение —
окно с остаточным временем и «Вставить», закрываемое по Esc.

Решение пользователя лечит симптом; корень лечится как в Wispr Flow —
вставкой в то окно, где диктовка началась. Утверждённый объём: возврат
фокуса + остаточное время на пилюле + Esc-отмена вставки.

## Компонент 1: вставка в исходное окно (ядро)

### Платформенный слой (`molvi/platform/win32/typer.py`, `molvi/platform/darwin/typer.py`)

Три новые публичные функции в обоих модулях:

- `get_target()` — «цель вставки» на момент вызова.
  Win: HWND активного окна (`GetForegroundWindow`); может быть 0 — тогда
  `None`. Mac: frontmost-приложение (`NSWorkspace.frontmostApplication`,
  храним pid) — точность **до приложения, не окна**: платформенное
  ограничение, фиксируем осознанно.
- `target_is_foreground(target) -> bool` — цель всё ещё активна.
- `activate_target(target) -> bool` — вернуть цели фокус; `True`, если после
  активации цель действительно стала активной (проверка обязательна:
  `SetForegroundWindow` на Windows может молча не сработать из-за
  foreground-lock; обход — `AttachThreadInput` к потоку целевого окна,
  детали — в плане). Окно закрыто/невалидно → `False` без исключения.
  После успешной активации — короткая пауза (~0.15 с), чтобы окно успело
  принять фокус до Ctrl+V.

### Controller (`molvi/controller.py`)

- Конструктор получает необязательный набор функций цели
  (`target_fns=(get_target, target_is_foreground, activate_target)` или
  `None` — старое поведение; тесты передают фейки).
- `on_release`: перед постановкой задания снять цель `get_target()` и
  положить в очередь кортеж `(audio, target)`.
- `_run`, после успешного `transcribe` и сохранения `_last_text`,
  перед `_insert_fn`:
  - `target is None` или `target_is_foreground(target)` → вставка как сейчас;
  - иначе `activate_target(target)`: успех → вставка; неудача →
    **вставку пропускаем** (вслепую в чужое окно не печатаем), уведомление
    `controller.target_lost` — текст уже в `last_text`, трей спасёт.

## Компонент 2: остаточное время на пилюле

- **Оценка, не прогресс** (faster-whisper прогресс не репортит): controller
  замеряет время каждого `transcribe` и держит скользящее среднее RTF
  (время обработки / длительность аудио; длительность = `len(audio) /
  samplerate`). Среднее живёт в памяти контроллера (не в Transcriber:
  не трогаем оба платформенных транскрайбера ради секундомера);
  сглаживание — экспоненциальное (новое значение с весом 0.5), сбрасывается
  при перезагрузке модели (`finish_model_reload` с новым transcriber).
- `Overlay.show_transcribing(eta_sec=None)` — новый необязательный параметр
  (существующие вызовы без него валидны). Пилюля в состоянии «Распознаю…»
  рисует убывающий счётчик `tr("overlay.eta", sec=N)` («~N с») в левой части
  пилюли на месте точки-индикатора (в жёлтом состоянии точка
  малоинформативна); тикает в существующем `_animate` от дедлайна
  `time.monotonic() + eta_sec`. Дошёл до нуля, а обработка идёт —
  показываем «~0 с», в минус не уходим.
- Первая диктовка после запуска (RTF ещё нет) → `eta_sec=None`, счётчик
  не рисуется, поведение пилюли как сейчас.

## Компонент 3: Esc — отмена вставки

- `HotkeyListener` (оба платформенных `hotkey.py`) получает необязательный
  колбэк `on_esc`: хук и так видит все клавиши; на нажатии Esc
  (VK 0x1B / keycode 53) зовём колбэк. Esc **не глотаем** — событие идёт
  дальше в активное окно. Компромисс осознанный: глотание ломало бы Esc
  в чужих окнах при фоновой обработке; двойное действие (Esc закрыл что-то
  в активном окне И отменил вставку) считаем меньшим злом — окно
  уязвимости только на время обработки.
- `Controller.cancel_pending()`: помечает текущее/ожидающие задания
  «не вставлять». Распознавание доводим до конца (текст сохраняется в
  `_last_text` — Esc отменяет вставку, не работу), вставку пропускаем,
  уведомление `controller.paste_cancelled`. Вне обработки (нет заданий)
  вызов — no-op, Esc в обычной работе ни на что не влияет: фильтрация
  состоянием контроллера, хук всегда репортит.
- Флаг отмены сбрасывается при следующем `on_release` (новая диктовка —
  новое намерение вставить).

## i18n (`molvi/i18n.py`)

Новые ключи (ru / en), паритет обязателен:

- `overlay.eta` — «~{sec} с» / "~{sec}s"
- `controller.target_lost` — «Не удалось вернуться в исходное окно —
  текст в трее («Скопировать последний текст»)» / "Couldn't switch back to
  the original window — the text is in the tray (\"Copy last transcript\")"
- `controller.paste_cancelled` — «Вставка отменена — текст в трее
  («Скопировать последний текст»)» / "Paste cancelled — the text is in the
  tray (\"Copy last transcript\")"

## Поток данных

```
hotkey release → get_target() → очередь (audio, target)
worker: transcribe (замер RTF) → _last_text → [cancel? → notify, стоп]
        → target активна? → вставка
        → сменилась? activate_target: успех → вставка; неудача → notify
show_transcribing(eta_sec = длительность_аудио × RTF | None)
Esc (хук) → controller.cancel_pending()
```

## Обработка ошибок

- `get_target()` упал/вернул None → работаем по-старому (вставка в текущее
  окно), фича деградирует тихо.
- `activate_target` бросил исключение → как `False`: пропуск вставки +
  `controller.target_lost`.
- Ошибки счётчика ETA не роняют анимацию пилюли (по образцу
  `_level_source`: try/except вокруг чтения).
- Esc-колбэк из потока хука не блокируется (только выставление флага под
  локом — как on_press/on_release).

## Тесты

1. Controller: цель снимается в `on_release` и едет с заданием; смена
   фокуса → вызван `activate_target`, вставка после успеха.
2. Controller: `activate_target` вернул `False` → вставки нет, уведомление
   `target_lost`, текст в `last_text`.
3. Controller: фокус не менялся → `activate_target` не вызывается, вставка
   как раньше.
4. Controller: `cancel_pending` во время обработки → вставки нет,
   уведомление `paste_cancelled`, текст в `last_text`; следующая диктовка
   вставляется (флаг сброшен).
5. Controller: RTF-среднее считается и даёт `eta_sec` второму заданию;
   первое задание — `eta_sec=None`; сброс при `finish_model_reload`.
6. Overlay: расчёт текста счётчика от дедлайна (включая пол «~0 с»);
   `show_transcribing()` без аргумента — прежнее поведение.
7. Hotkey (win/darwin): нажатие Esc зовёт `on_esc`; Esc не входит в комбо
   и не глотается; без колбэка — поведение прежнее.
8. Платформенные (win-only, моки WinAPI): `activate_target` возвращает
   `False` при невалидном окне; проверка активации после
   `SetForegroundWindow`.

## Отвергнутые альтернативы

- **Окно с кнопкой «Вставить» (предложение пользователя буквально)** —
  авто-возврат фокуса убирает саму необходимость ручной кнопки; кнопка —
  лишний клик в 100% диктовок ради контроля, нужного в 1%. Esc оставлен
  как аварийный тормоз.
- **Глотать Esc во время обработки** — сломало бы Esc в другом окне, где
  пользователь работает, пока идёт фоновая обработка.
- **Прерывать распознавание по Esc** — прерывание генератора faster-whisper
  сложнее и лишает текста (в трее его бы не было); доводим до конца.
- **RTF в Transcriber** — пришлось бы дублировать в двух платформенных
  транскрайберах; секундомер — забота оркестратора.
- **Хранить HWND конкретного контрола (не окна)** — фокус внутри окна
  восстанавливает само приложение; точность до окна достаточна.

## Ограничения (фиксируем честно)

- Возврат фокуса на Windows может не сработать (foreground-lock) — fallback
  через трей, текст не теряется.
- macOS: возврат к приложению, не к конкретному окну.
- ETA — оценка по прошлой скорости, не реальный прогресс.
- Esc двойного назначения: уходит и в активное окно.
