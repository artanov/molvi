# Спека: «Скопировать последний текст» в трее

Дата: 2026-07-20

## Проблема

Вставка распознанного текста идёт через буфер обмена: текст кладётся в буфер,
эмулируется Ctrl+V (Cmd+V на маке), через 0.3 с буфер восстанавливается к
прежнему содержимому (`platform/win32/typer.py:paste_text`,
`platform/darwin/typer.py:paste_text`). Ctrl+V уходит в окно, которое в фокусе
в момент окончания обработки. Если пользователь за время распознавания (при
длинной диктовке — десятки секунд) переключился в другое окно, вставка попадает
не туда или «в никуда», а восстановление буфера затирает текст. В приложении
текст нигде не сохраняется — расшифровка теряется безвозвратно.

Фидбек пользователей: «напрямую в буфер писать не очень каждый запрос, а вот
через специальную кнопочку вернуть утраченный текст будет очень полезно».

## Решение

Хранить последнюю расшифровку в памяти и добавить в меню трея пункт
«Скопировать последний текст». Только последняя диктовка, без истории, без
сохранения на диск, без эвристик определения потери фокуса (осознанно
отвергнуты — см. «Отвергнутые альтернативы»).

## Компоненты

### Controller (`molvi/controller.py`)

- Новое поле `self._last_text = None`.
- В `_run()` сразу после успешного `transcribe()` — до попытки вставки —
  сохранять текст под `self._lock`: расшифровка не должна пропасть, даже если
  вставка упадёт.
- Новый метод `last_text()` — возвращает сохранённый текст (под локом),
  `None`, если диктовок ещё не было. Пустые расшифровки (`text` falsy)
  не перезаписывают сохранённое.

### Платформенный слой (`molvi/platform/win32/typer.py`, `molvi/platform/darwin/typer.py`)

- Публичная функция `copy_to_clipboard(text)` в обоих модулях — тонкая обёртка
  над существующим `_set_clipboard_text()`. Без эмуляции Ctrl+V и без
  восстановления буфера: пользователь явно попросил положить текст в буфер.

### Tray (`molvi/tray.py`)

- Новый пункт меню между «Настройки» и «Пауза»: `tr("tray.copy_last")`.
- Новый конструкторный коллбэк `on_copy_last` (по образцу `on_settings`,
  с дефолтом-заглушкой).
- Пункт неактивен, пока текста нет: `pystray.MenuItem(..., enabled=...)`
  с callable, дёргающим второй коллбэк `has_last_text` (тоже параметр
  конструктора, дефолт `lambda: False`).

### Связка (`molvi/app.py`)

- Коллбэки передаются в `Tray(...)` с защитой от `controller is None`
  (трей стартует до загрузки модели, как у `on_toggle_pause`):
  - `has_last_text`: `controller is not None and controller.last_text() is not None`
  - `on_copy_last`: взять `controller.last_text()`, вызвать
    `typer.copy_to_clipboard()`, показать `tray.notify(tr("app.notify.copied"))`.
    При исключении (буфер занят, `_open_clipboard` уже ретраит внутри) —
    `tray.notify(tr("app.notify.copy_failed", exc=exc))`, трей не роняем.

### i18n (`molvi/i18n.py`)

Новые ключи (ru / en):

- `tray.copy_last` — «Скопировать последний текст» / "Copy last transcript"
- `app.notify.copied` — «Текст скопирован в буфер обмена» / "Text copied to clipboard"
- `app.notify.copy_failed` — «Не удалось скопировать: {exc}» / "Copy failed: {exc}"

## Поток данных

```
hotkey release → transcribe → [сохранить в _last_text] → insert_text (как раньше)
трей: клик «Скопировать последний текст» → controller.last_text()
      → typer.copy_to_clipboard() → notify «скопировано»
```

## Обработка ошибок

- Вставка упала → текст уже сохранён в `_last_text`, существующее уведомление
  об ошибке вставки остаётся как есть.
- Клик по пункту при гонке (текст ещё None, меню не обновилось) → тихий no-op.
- Ошибка записи в буфер → уведомление, без падения трея.

## Тесты

1. `Controller`: после успешной расшифровки `last_text()` возвращает текст.
2. `Controller`: текст сохраняется, даже если `insert_fn` бросил исключение.
3. `Controller`: `last_text()` — `None` до первой диктовки; пустая расшифровка
   не затирает предыдущий текст.
4. `Tray`: пункт меню вызывает `on_copy_last`; `enabled` отражает
   `has_last_text`.
5. win-only (skip на других ОС): `copy_to_clipboard` кладёт текст в буфер.

## Отвергнутые альтернативы

- **История N диктовок / сохранение на диск** — лишний UI и вопросы
  приватности; последняя диктовка закрывает основной сценарий. Можно добавить
  позже, если попросят.
- **Эвристика «окно сменилось → уведомить»** — ненадёжна (смена вкладки внутри
  окна не ловится) и добавляет шум; пользователь выбрал минимальный вариант.
- **Хранить текст в `Tray`** — трей это чистый UI, состояние живёт
  в `Controller`.
- **Всегда оставлять текст в буфере после вставки** — ломает
  clipboard-workflow пользователя, о чём пользователи прямо пишут.
